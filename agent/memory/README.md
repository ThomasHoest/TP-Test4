# Memory

The assistant's persistent, curated knowledge. **Curated** is the
operative word: this folder holds distilled, decision-relevant
content — not raw transcripts. Per-person records live separately in
[`../../user/people/`](../../user/people/).

## Structure

```
agent/memory/
├── README.md          (this file — schema + lifecycle)
├── taxonomy.md        (the routing registry — category → destination → shape)
├── index.md           (the memory map — a system-generated projection; do not edit)
├── decisions.md       (running ledger of explicit decisions)
├── opportunities.md   (live pipeline — leads, gigs, possibilities)
├── observations.md    (patterns noticed about you)
├── priorities.md      (your priorities tracker, surfaced in digests)
├── change-log.jsonl   (append-only audit of every memory change; system-written)
├── topics/            (one file per topic — accumulated understanding)
├── short-term-memory/ (per-channel session transcripts)
│   └── web-cli/       (the web CLI channel; see its README)
├── log/               (chronological daily / weekly markers)
│   └── README.md
└── skill_usage.md     (every skill invocation + observed gaps — mandatory)
```

Files except `README.md`, `taxonomy.md`, and `index.md` are created on
demand — they don't ship empty in the template. The first time the
assistant has a decision to record, it creates `decisions.md`; same for
the others. `change-log.jsonl` is created on the first memory change.

### `taxonomy.md` + `index.md` — the living memory map

`taxonomy.md` is a small markdown table mapping each **category** of
information to its **destination** file and **shape**. `index.md` (the
"memory map") is a *projection* of that table over the repo's actual
contents, regenerated by the system at the end of every turn —
**read it first to orient, never hand-edit it.** To evolve the
structure (add a category, move a destination), edit `taxonomy.md`; the
map follows automatically, no code change. A file whose front-matter
`type:` isn't in the table surfaces in the map as an "unregistered
type" — a prompt to add it to `taxonomy.md`.

## Front-matter conventions

Memory files self-describe with a small YAML front-matter block so the
map can project them generically. Keys (all optional except `type` on
typed files):

```yaml
---
type: person          # the taxonomy category this file belongs to (drives routing/orientation)
date: 2026-06-16      # the date the entry is *about* (not necessarily creation)
confidence: high      # high | medium | low — for inferred/observed content
status: active        # active | archived — archived content drops out of the hot slice
---
```

`type` is what links a file to a `taxonomy.md` row; keep it consistent
with the registry's category names.

## When the assistant writes here

| Trigger | Where it lands |
|---|---|
| New person enters your orbit | `user/people/{slug}.md` |
| A tracked topic shifts | append/update `topics/{topic}.md` |
| You make an explicit decision | append to `decisions.md` |
| An opportunity moves (new lead, status change) | update `opportunities.md` |
| The assistant notices a pattern across multiple days | append to `observations.md` |
| **Any skill is invoked** | append row to `skill_usage.md` (mandatory) |
| The assistant needed a skill that didn't exist | append gap entry to `skill_usage.md` |
| End of day | summary in `log/{YYYY}/{MM}/{YYYY-MM-DD}.md` |

## What does NOT go here

- Raw chat transcripts → those go in `short-term-memory/<channel>/`.
- Speculation the assistant hasn't validated against you.
- Anything sensitive about other people that you haven't authorized.
- Content that's better served by reading the source on demand.

## File schemas

### `people/{slug}.md` (canonical schema)
```markdown
---
name: Full Name
slug: full-name
role: current role
company: current company
relationship: how you know them
priority: must-flag | regular | background
cadence: weekly | biweekly | monthly | quarterly | opportunistic | dormant
last_touch: YYYY-MM-DD
fit: high | medium | low | unknown | n/a
email: preferred@example.com
linkedin: handle-or-url
---

## Context
Who they are; how you know them; what they care about.

## Suggested follow-up
One of: accept + personal message · accept + observe · defer · decline.

## Recent
Append-only, newest at top.
- {YYYY-MM-DD} · {email | call | meeting | message | inferred} · summary; agreed next step.
```

Files themselves live in [`../../user/people/`](../../user/people/);
maintained by [`../skills/track-network.md`](../skills/track-network.md).

### `topics/{topic}.md`
```markdown
# {Topic}

## Current understanding
What is the state of this topic right now, in your mental model.

## Recent shifts
- {YYYY-MM-DD}: what changed and why it matters.

## Open questions
What is unresolved.

## Sources of record
Where the canonical view of this topic comes from.
```

### `decisions.md` — append-only ledger
```markdown
## {YYYY-MM-DD}: {decision}
- Context:
- Decision:
- Rationale:
- Implications:
```

### `opportunities.md` — pipeline
```markdown
## {Opportunity name}
- Status: lead | qualifying | active | dormant | won | lost
- Source: how it surfaced
- Match: why this fits (or doesn't)
- Last update: {YYYY-MM-DD}
- Next move:
```

### `observations.md` — append-only
```markdown
## {YYYY-MM-DD}: {observation}
What was noticed; how confident; whether to act on it.
```

### `log/{YYYY}/{MM}/{YYYY-MM-DD}.md`
Daily marker — what the assistant ran, what surfaced, what you
reacted to. See [`log/README.md`](log/README.md) for the structure.

## Hygiene

- **Pruning.** Old `log/` entries roll up monthly into a summary
  entry; older dailies can be archived. Pattern: keep the last 60
  days of daily logs hot; older folds into monthly rollups.
- **Conflicts.** When two memory entries disagree (e.g. someone
  changed roles), keep the latest authoritative entry and note the
  change in the file's `## Recent` section.
- **Decay.** Memory should serve current decisions. If an entry
  hasn't been touched or referenced in 6 months and isn't
  load-bearing, mark it `archived` rather than deleting.
